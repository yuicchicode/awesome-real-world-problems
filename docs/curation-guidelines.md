# Curation Guidelines

This repository is a curated collection of **real-world problems** that can inspire meaningful portfolio projects and open-source solutions.

The goal is not to collect generic app ideas.
The goal is to document real pain points in a way that feels human, descriptive, and useful for builders.

---

## What this repository is for

This repository exists to help developers explore problems that are:

- grounded in real life
- useful for open-source collaboration
- relevant to actual people, teams, or communities
- rich enough to support technical exploration
- broader and more meaningful than “build a basic app”

Each entry should help someone understand:
- what the problem is
- how it feels in real life
- why it keeps happening
- why it matters
- what kinds of technical directions could help solve it

---

## What belongs here

A good entry usually describes a problem that:

- exists in real life, not only in theory
- affects a person, team, family, business, clinic, school, NGO, or community
- creates friction, waste, confusion, risk, delay, stress, or poor decisions
- is rich enough to inspire a real product or workflow
- can be approached through software, data, automation, coordination, or visibility

Examples of strong entries:
- Difficulty tracking the patient’s progress outside the clinic
- Parents lack clear visibility into the student’s real progress
- No inventory control
- Delivery delays
- Poorly organized volunteers

---

## What does not belong here

This repository is **not** meant for:

- generic app ideas
- startup buzzword ideas
- feature lists without a real problem behind them
- vague innovation concepts
- solution-first entries with no real context
- “build X clone” style suggestions

Weak examples:
- Build a to-do app
- Create a CRM
- Make a scheduling platform
- Build an AI assistant for everything
- Create a logistics dashboard

Those may become valid products, but they are not problem statements by themselves.

---

## Editorial principles

### 1. Problem first
Entries should describe the pain before describing the solution.

Prefer:
- Lead information is scattered across multiple platforms

Avoid:
- Build a lead dashboard

---

### 2. Real-world tone
Entries should feel like they came from real experience.

The best entries sound like:
- someone actually dealing with the problem
- someone trying to keep things working through memory, messages, spreadsheets, or improvisation
- someone frustrated by lack of visibility, coordination, or continuity

---

### 3. Human before technical
Each entry should begin with a human explanation of the problem.

The technical layer should come later, usually in sections like:
- Why this is difficult
- Open-source opportunity
- Technical requirements
- Suggested MVP

---

### 4. Descriptive, not inflated
Use clear, concrete language.

Prefer:
- The owner ends up doing everything alone

Avoid:
- Local commerce faces operational centralization challenges in low-structure environments

---

### 5. Practical open-source relevance
A good entry should make people think:
- this is real
- this matters
- I can build something around this
- this could help other people too

---

## Recommended structure for each problem

Each problem entry should follow this format:

1. Problem title
2. Problem summary
3. Real-world report
4. Why this is difficult
5. Why it matters
6. Open-source opportunity
7. Technical requirements
8. Suggested MVP
9. Possible extensions

This format helps keep entries readable, human, and useful for builders.

---

## Writing the “Real-world report”

This is the most important section.

It should:
- be written in first person
- sound natural
- describe how the problem happens in real life
- show what depends on memory, improvisation, or fragmented communication
- make the pain visible

Good signals in this section:
- “In practice...”
- “Most of the time...”
- “The problem is that...”
- “Even when I try to do it well...”
- “This creates confusion because...”

This section should not sound like marketing copy.

---

## Writing the technical section

The technical section should not lock contributors into only one solution.

It should help them think in terms of:
- roles
- entities
- workflows
- recurring tasks
- alerts
- timelines
- dashboards
- integrations
- status tracking
- constraints
- auditability
- mobile use
- incomplete data

Good technical requirements are:
- practical
- implementation-friendly
- broad enough to support multiple solution directions

---

## Category fit

A problem should be placed in the category where its **main operational context** fits best.

Some overlap is normal.

Examples:
- A child’s school routine belongs more naturally in `daily-life` or `education` depending on the focus
- A patient follow-up problem belongs in `healthcare`
- A multi-party customs and shipment visibility problem belongs in `international-trade`

Choose the category that best reflects the core context of the problem.

---

## Avoiding duplicates

Before adding a new entry:
- check whether the same problem already exists
- check whether the same idea appears with slightly different wording
- check whether the problem belongs in a neighboring category instead

Not every similar problem is a duplicate, but entries should not repeat the same pain with only cosmetic wording changes.

---

## Language

The main language of the repository is **English**.

Problem entries should be written in English first.

Portuguese can exist in supporting documentation, category pages, or future translations, but the canonical problem statement should be clear and readable in English.

---

## Quality checklist

A strong entry usually does all of the following:

- describes a real problem
- sounds human and grounded
- explains why the problem is difficult
- explains why the problem matters
- connects the pain to open-source potential
- includes practical technical requirements
- leaves room for multiple implementations
- avoids solution-first language
- avoids shallow generic ideas

---

## Final rule of thumb

A good entry should make a contributor think:

> “I understand the pain, I understand why it matters, and I can already imagine a useful open-source project around it.”
