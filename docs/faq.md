# FAQ

## What is this repository?
This repository is a curated collection of real-world problems that developers can use as inspiration for meaningful portfolio projects and open-source solutions.

It focuses on problems, not generic app ideas.

---

## Why is this repository not just a list of app ideas?
Because generic app ideas are usually too shallow to reflect real-world work.

This project tries to document actual pain points, workflows, coordination failures, visibility problems, and operational gaps that people deal with in real life.

---

## Why are the problems written in first person?
Because first-person writing makes the problem feel real.

It helps contributors understand the human side of the issue before jumping into features, architecture, or implementation.

---

## Is this repository only for open-source developers?
No.

It is mainly designed for developers who want to build better portfolio projects or open-source tools, but the problem statements can also be useful for product thinkers, designers, students, and technical writers.

---

## Can I contribute a new problem?
Yes.

A good contribution should describe a real-world problem clearly, realistically, and with enough context to help others understand why it matters.

Start by reading:
- `CONTRIBUTING.md`
- `docs/curation-guidelines.md`

---

## Can I contribute a solution instead of a problem?
This repository is problem-first.

That means the main goal is to document the pain clearly, not to turn each entry into a finished solution spec.

You may suggest technical directions, MVP ideas, and extensions inside the entry, but the core of the contribution should still be the problem itself.

---

## Can one problem belong to more than one category?
Sometimes, yes.

But each entry should be placed in the category that best matches its main context.

Try to choose the category based on where the problem is most naturally experienced.

---

## Does every problem need technical requirements?
Yes, ideally.

One of the goals of this project is to help people move from a real-world pain point to something buildable.

Technical requirements help contributors think in terms of workflows, roles, entities, constraints, reminders, timelines, dashboards, and integrations.

---

## Does every problem need to be very technical?
No.

The entry should start human first.

The technical part exists to help builders later, but the problem itself should still be understandable to someone reading it as a lived experience.

---

## Why are the entries so descriptive?
Because real problems are usually not simple.

The extra context helps contributors understand:
- why the problem exists
- what makes it hard
- what usually goes wrong
- why solving it is meaningful

---

## Are generic project ideas accepted?
No, not as standalone entries.

Examples that do not fit well:
- build a to-do app
- create a CRM
- build a school platform
- make a delivery dashboard

These are too solution-first and too broad without real context.

---

## What makes a problem a good fit for this repository?
Usually, a strong entry has:
- a real-world context
- clear pain
- practical impact
- room for multiple implementations
- technical depth
- open-source relevance

---

## Does the repository prefer English or Portuguese?
The main language is English.

English is the canonical language for the problem entries, since the repository is meant to be broadly accessible.

Portuguese can appear in supporting documentation, category pages, and other complementary materials.

---

## Can I translate content?
Yes.

Translations are welcome, especially when they preserve the original meaning and keep the same descriptive tone.

---

## What kinds of problems are most valuable here?
Problems that involve:
- fragmented information
- weak visibility
- recurring coordination
- memory-based workflows
- manual follow-up
- overloaded people
- inconsistent routines
- avoidable delays
- real operational pain

---

## Is this repository only for SaaS-like ideas?
No.

A problem can be solved through many different formats:
- internal tools
- community tools
- dashboards
- workflow systems
- reminder systems
- mobile apps
- data tools
- offline-friendly software
- coordination platforms

---

## Why does the repository include “Suggested MVP” and “Possible extensions”?
Because contributors often need help imagining how a real pain point can become a buildable project.

Those sections are there to guide thinking, not to lock the project into one product direction.

---

## I have a problem idea, but I am not sure it fits. What should I do?
Compare it against these questions:

- Is this a real pain point?
- Does it affect actual people, teams, or communities?
- Does it go beyond a generic app idea?
- Can someone imagine multiple possible solutions?
- Does it have meaningful open-source value?

If the answer is mostly yes, it may be a good fit.

---

## Why are there category READMEs?
Each category README explains the kind of problems that belong in that area.

They help contributors navigate the repository and avoid placing entries in categories that do not match the problem’s real context.

---

## What is the long-term goal of the repository?
The long-term goal is to become a useful, community-driven collection of real-world problems that inspire stronger portfolio projects, more relevant open-source work, and better developer creativity grounded in reality.
